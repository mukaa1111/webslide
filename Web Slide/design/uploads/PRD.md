## 1. 프로젝트 개요

### 1.1 비전
> **"영상보다 빠른 지식 소비(Faster Knowledge Consumption)"** — 쌓이는 영상 콘텐츠를 텍스트·슬라이드로 빠르게 흡수하고, 검색 가능한 개인 지식 자산으로 영구 축적한다.

### 1.2 배경 및 기획 동기

**핵심 인사이트**: 영상으로 내용을 소비하는 것보다 텍스트·슬라이드 형식으로 읽는 것이 핵심 파악 속도가 훨씬 빠르다.

**문제 상황**:
- 유튜브 기술 영상, 컨퍼런스 발표, 강의 영상이 끊임없이 쌓인다.
- 영상 1시간을 보려면 1시간이 필요하지만, 같은 내용을 슬라이드로 읽으면 3~10분이면 핵심 파악이 가능하다.
- "나중에 보려고" 저장한 영상은 결국 소비되지 못하고 플레이리스트에 묻힌다.

**솔루션 흐름**:
```
[사용자 수동 단계]
영상(YouTube, 컨퍼런스 등)
    │
    ▼
NotebookLM에 영상 URL 투입
    │ → 주요 내용 요약·구조화 → .md 파일 생성
    ▼
로컬 /inputs 폴더에 .md 저장

[자동화 단계 — 본 파이프라인]
python run.py 실행
    │
    ▼
6-에이전트 파이프라인 (Scanner → ... → Evaluator)
    │
    ▼
Supabase knowledge_slides 저장
    │
    ▼
Next.js 슬라이드 뷰어 (Vercel)
→ 언제 어디서든 빠르게 탐색·검색·재활용
```

**NotebookLM의 역할**: 사용자가 원하는 영상을 직접 NotebookLM에 입력하면, NotebookLM이 영상의 핵심 내용을 구조화된 마크다운으로 정리해 준다. 이 결과물(.md)을 사용자가 로컬 `/inputs` 폴더에 저장하는 것이 파이프라인의 입력 데이터가 된다.

### 1.3 MVP 범위

**포함 (In-Scope)**
- `/inputs` 폴더 .md 파일 전수 처리 (배치 1회 실행)
- 6-에이전트 파이프라인 (Scanner → Parser → Validator → Repair → Uploader → Evaluator)
- Supabase `knowledge_slides` + `pipeline_log` 테이블 저장
- Next.js 슬라이드 뷰어 (Vercel 배포)
- 기본 과목별 필터 + 태그 검색 UI

**제외 (Out-of-Scope, Phase 2+)**
- 실시간 파일 감시 (watchdog)
- 사용자 인증 / 권한 관리
- AI 기반 자동 태깅·요약
- LMS 연동 (수강 이력, 수료증)
- 학습자 진도 퍼시스턴스 (서버 저장)

## 3. 기능 요구사항

### 3.1 파이프라인 핵심 기능

#### FR-01: 파일 스캔 및 큐 생성
- `/inputs` 폴더의 `NN_제목.md` 형식 파일을 자연어 정렬로 처리
- `pipeline_state.json`의 `processed` 목록과 대조하여 중복 처리 방지
- 스캔 결과를 콘솔에 실시간 출력

#### FR-02: 마크다운 → 슬라이드 JSON 변환
- YAML frontmatter 파싱 (`python-frontmatter`)
- `---` 구분자 기준 슬라이드 분리, 없을 시 H2(`##`) 기준 폴백
- 레이아웃 자동 감지 (5종: default, code-only, bullet-list, image-focus, title-only)
- SHA256 해시로 변경 감지 (`raw_md_hash`)

#### FR-03: 자동 복구 (최대 2회)
- Parser/Validator 실패 시 `repair_agent` 자동 호출
- 복구 전략: 파일명 기반 메타 생성, H2 재분리, slug 변환, 최소 유효 데이터 생성
- 2회 실패 시 `/quarantine` 이동 + JSON 격리 리포트 생성

#### FR-04: Supabase upsert
- `UNIQUE(source_file, slide_index)` 제약으로 멱등성 보장
- `tenacity` 3회 지수 백오프 재시도 (2→4→8초)
- 원본 .md를 Storage `md-backups` 버킷에 백업

#### FR-05: 무결성 검증 및 완료 처리
- DB SELECT로 파싱 수 vs 저장 수 비교 (허용 오차 0)
- 불일치 시 누락 슬라이드만 선택적 재업로드 1회
- 검증 통과 시 `/processed`로 파일 이동
- `pipeline_log` 테이블에 처리 기록

#### FR-06: 상태 영속화 및 재시작
- `pipeline_state.json`에 매 파일 완료 후 상태 저장
- 재실행 시 `processed` 목록 로드 후 미처리 파일만 처리
- 최종 실행 요약 리포트를 `/state/run_report_{run_id}.json`에 저장

### 3.2 프론트엔드 UX 기능

#### FR-07: 슬라이드 뷰어 — 탐색
- 키보드 ← → (또는 PageUp/PageDown) 슬라이드 이동
- 터치/스와이프 지원 (모바일 `Hammer.js` 또는 `use-swipe` 훅)
- URL 파라미터로 특정 슬라이드 딥링크 가능 (`?slide=3`)

#### FR-08: 슬라이드 뷰어 — 진행률 표시
- 상단 Progress Bar (현재 슬라이드 / 전체 슬라이드)
- "3 / 12" 형식의 텍스트 카운터 병기
- 레이아웃별 색상 코딩 (code-only: 남색, bullet-list: 녹색 등 선택적)

#### FR-09: 코드 블록 렌더링
- `Shiki` 또는 `Prism.js` 기반 Syntax Highlight
- 언어 자동 감지 (frontmatter `lang` 필드 우선, 없으면 자동 감지)
- 코드 복사 버튼 (클립보드 API)
- 다크/라이트 테마 토글

#### FR-10: 발표자 노트 패널
- 슬라이드 하단 "Notes" 토글 버튼
- 접기/펼치기 애니메이션 (Framer Motion 또는 CSS transition)
- 프린트 모드에서는 자동 펼침

#### FR-11: 과목/태그 탐색 (홈 화면)
- 코스별 카드 목록 (`/` 루트 페이지)
- 태그 클라우드 필터 (multi-select)
- 검색 입력창 (제목 + 내용 기반 클라이언트사이드 퍼지 검색)
- 레벨 배지 (beginner/intermediate/advanced 색상 구분)

#### FR-12: 반응형 레이아웃
- 모바일: 단일 컬럼, 풀스크린 슬라이드
- 태블릿: 사이드바 TOC + 슬라이드 영역
- 데스크톱: 3-패널 (과목 목록 | 슬라이드 | 노트)

## 4. 비기능 요구사항

### 4.1 성능 (KPI)

| 지표 | 목표값 | 측정 방법 |
|------|--------|----------|
| **파싱 성공률** | ≥ 95% | `success / total × 100` (run_report) |
| **자동 복구율** | ≥ 80% (복구 가능 오류 중) | `repaired / (repaired + quarantined)` |
| **업로드 무결성** | 100% | evaluator: `expected == actual` |
| **단일 파일 처리 시간** | ≤ 5초 (10슬라이드 기준) | pipeline_state timestamp diff |
| **배치 처리 속도** | ≤ 1분 (10파일 기준) | run_report.duration_seconds |
| **슬라이드 초기 로드** | ≤ 1.5초 P95 | Vercel Analytics / Lighthouse |
| **검색 응답 속도** | ≤ 300ms | 클라이언트사이드 필터 기준 |

### 4.2 신뢰성

| 조건 | 요구사항 |
|------|---------|
| **멱등성** | 동일 파일 재실행 시 DB 중복 없이 UPDATE만 발생 |
| **격리 보장** | 단일 파일 오류가 배치 전체를 중단시키지 않음 |
| **재시작 안전성** | 중간 중단 후 재실행 시 이미 처리된 파일 건너뜀 |
| **FATAL 최소화** | DB 연결 오류만 FATAL — 그 외 모든 오류는 스킵·격리 처리 |

### 4.3 보안

| 항목 | 요구사항 |
|------|---------|
| **API 키 관리** | `SUPABASE_SERVICE_ROLE_KEY`는 `.env`에만, `.gitignore` 필수 |
| **클라이언트 노출 금지** | Service Role Key는 파이프라인 서버에서만 사용 |
| **프론트엔드 키** | `SUPABASE_ANON_KEY` + RLS 정책으로 read-only 제한 |
| **Storage 접근** | `md-backups` 버킷은 private (서비스 롤만 접근) |

## 5. 예외 상황 방어 로직 (비판적 검토)

파이프라인 운영 중 발생 가능한 **예측 예외 시나리오**와 방어 전략:

### 5.1 Scanner 레이어 예외

| 예외 | 방어 로직 |
|------|----------|
| **빈 파일 (0 bytes)** | `MIN_FILE_SIZE = 10` bytes 체크 → WARN + 스킵 |
| **용량 초과 파일** | `MAX_FILE_SIZE = 5MB` 상한 설정 → WARN + 스킵 (현재 미정의, 추가 필요) |
| **파일명 형식 불일치** | `NN_제목.md` 패턴 미매칭 시 WARN + 처리는 계속 (stem을 파일명 전체로 사용) |
| **숨김 파일 포함** | `.`으로 시작하는 경로 컴포넌트 필터링 |
| **심볼릭 링크** | `follow_symlinks=False`로 스캔, 실제 파일만 처리 |
| **동일 파일명 중복** | 자연어 정렬 후 중복 stem 감지 → WARN (후순위 파일 스킵) |
| **인코딩 감지 실패** | `chardet` + 5단계 폴백 (utf-8-sig → utf-8 → cp949 → euc-kr → latin-1) |

### 5.2 Parser 레이어 예외

| 예외 | 방어 로직 |
|------|----------|
| **frontmatter 없음** | 파일명 기반 메타 자동 생성 (`auto-` prefix, `_auto_generated: true`) |
| **YAML 특수문자 오류** | regex 사전 정제 후 재파싱 → 실패 시 frontmatter 완전 제거 후 재시도 |
| **슬라이드 0개 (`---` 없음)** | H2(`##`) 기준 재분리 → H1 기준 → 전체 단일 슬라이드 병합 |
| **과도하게 긴 슬라이드** | 단일 슬라이드 본문 20,000자 초과 시 WARN (잘리지 않고 저장) |
| **중첩 YAML 오류** | `safe_load` 사용, 예외 발생 시 ErrorCode `FRONTMATTER_SYNTAX`로 repair 위임 |
| **이진 파일 오감지** | 내용 중 `\x00` null byte 존재 시 WARN + 스킵 |

### 5.3 Validator 레이어 예외

| 예외 | 방어 로직 |
|------|----------|
| **courseId 한글 포함** | slugify 변환 (한글 유지 방식 — `re.sub(r'[^\w가-힣-]', '')`) |
| **슬라이드 50개 초과** | WARN 발생하지만 업로드는 진행 (비즈니스 판단) |
| **duration 0 또는 None** | 기본값 주입 (슬라이드 수 × 2분) |
| **tags가 string으로 입력됨** | `"python,basics"` → `["python", "basics"]` 자동 분리 |

### 5.4 Uploader 레이어 예외

| 예외 | 방어 로직 |
|------|----------|
| **Supabase 504 Timeout** | `tenacity` 재시도 (2→4→8초), 3회 후 DEAD_LETTER |
| **인증 만료 (401)** | `fatal=True` → 즉시 중단, 재인증 안내 로그 |
| **Row-level Security 위반** | Service Role Key 확인 → `fatal=True` |
| **Storage 버킷 없음** | 버킷 자동 생성 시도 → 실패 시 WARN (Storage 백업 스킵, upsert는 계속) |
| **JSON 직렬화 실패** | `ParsedDocument` 내 직렬화 불가 타입 검사 후 `str()` 변환 fallback |
| **부분 upsert 성공** | 응답의 `data` 배열 길이로 실제 저장 수 확인 → evaluator에서 재검증 |

### 5.5 Network / 환경 예외

| 예외 | 방어 로직 |
|------|----------|
| **`.env` 파일 없음** | 실행 초기에 필수 환경변수 존재 확인 → 없으면 명확한 에러 메시지 후 종료 |
| **Supabase 프로젝트 일시정지** | HTTP 503 → `fatal=True` + 안내 메시지 |
| **디스크 쓰기 권한 없음** | `/processed`, `/quarantine` 디렉토리 권한 사전 확인 |

### 6.2 지식 검색 UI 구조 (Search Architecture)

#### 6.2.1 홈 화면 구조
```
/                          (과목 목록 + 검색)
├── 검색창 (상단 고정)
├── 태그 필터 (수평 스크롤 가능한 chip 목록)
├── 레벨 필터 (beginner / intermediate / advanced 토글)
└── 과목 카드 그리드
    ├── 썸네일 (첫 슬라이드 내용 미리보기)
    ├── 제목 + 슬라이드 수
    ├── 태그 배지 (최대 3개 + "더보기")
    └── 레벨 배지

/course/[courseId]         (슬라이드 뷰어)
/course/[courseId]?slide=N (특정 슬라이드 딥링크)
/search?q=[query]          (검색 결과 페이지)
```

#### 6.2.2 검색 전략 (MVP: 클라이언트사이드)
```
1. 초기 로드 시 전체 슬라이드 인덱스 fetch (제목 + 태그 + courseId)
   → /api/search-index 엔드포인트 (캐시: max-age 300s)

2. 퍼지 검색: Fuse.js (가중치: title 0.4 + tags 0.3 + courseId 0.3)

3. 결과 표시:
   - 슬라이드 제목 + 속한 코스명
   - 매칭 키워드 하이라이트
   - 클릭 시 해당 슬라이드로 직접 이동
```

## 8. 용어 정의

| 용어 | 정의 |
|------|------|
| **ParsedDocument** | parser_agent가 .md에서 변환한 구조화 JSON (슬라이드 배열 포함) |
| **SlideData** | 단일 슬라이드 객체 (index, title, layout, blocks, notes) |
| **ContentBlock** | 슬라이드 내 단위 콘텐츠 (paragraph, code, list, image 등) |
| **courseId** | 코스 식별자, slug 형식 (예: `python-fundamentals`) |
| **degraded** | repair_agent가 최소화 복구한 문서 (품질 저하 허용) |
| **dead_letter** | 3회 재시도 후에도 DB 업로드 실패한 파일 목록 |
| **raw_md_hash** | 원본 .md SHA256 앞 16자리, upsert 변경 감지에 활용 |
